# Plan: Raspberry Pi Health Dashboard on Analytics Server

## Context

The BBC Radio Player analytics server (`api/analytics_server.py`) runs on port **5002** on the Pi behind Nginx (port 5000). It already has:
- `GET /health` — basic health check returning `{"status": "ok"}`
- `GET /` — a full HTML status page with analytics stats (station/podcast popularity, index stats)
- Privacy-first design (no IPs, no PII)

The Pi also runs:
- Search server on port 5001 (`api/search_server.py`)
- Timer-based index builder and popular-export services
- Nginx reverse proxy on port 5000, with Tailscale Funnel providing HTTPS on :8443

## Goal

Add a **mobile-friendly health dashboard page** to the existing analytics server at `GET /health` that shows real-time Pi system metrics:
- Service status (analytics, search, index-builder timer, popular-export timer)
- CPU usage
- Memory usage
- Disk usage
- Uptime
- Temperature

The page will be accessible from a phone via the Tailscale URL (e.g. `https://raspberrypi.tailc23afa.ts.net:8443/health`).

## Approach: Extend the analytics server

Add a new **HTML health dashboard** to `api/analytics_server.py` at the existing `/health` route, plus a new `GET /health/api` JSON endpoint for the raw system metrics.

### Step 2: Add JSON health endpoint `GET /health/api`

A new endpoint returning structured JSON for all metrics:

```json
{
  "status": "ok",
  "timestamp": "...",
  "uptime": "5d 14h 23m",
  "cpu_percent": 12.5,
  "memory": { "total_mb": 4096, "used_mb": 1843, "available_mb": 2253, "percent": 45.0 },
  "disk": { "total_gb": 64, "used_gb": 22, "available_gb": 42, "percent": 34.4 },
  "temperature_c": 48.2,
  "services": {
    "analytics_server": {"status": "active", "port": 5002},
    "search_server": {"status": "active", "port": 5001},
    "index_builder_timer": {"status": "active", "next_run": "..."},
    "popular_export_timer": {"status": "active", "next_run": "..."}
  }
}
```

Implementation:
- Add route `@app.route('/health/api', methods=['GET'])`
- All metric collection wrapped in try/except so one failure doesn't break the endpoint
- Each metric returns `"unavailable"` on error

### Step 3: Replace the `/health` HTML page with a mobile-friendly dashboard

Replace the existing `/health` endpoint (`return jsonify({'status': 'ok'})`) with an HTML page when the `Accept` header prefers HTML (or when accessed from a browser). The page will:

- Use the existing CSS design language (same font, panel style, dark mode support)
- Be fully responsive and mobile-first
- Auto-refresh every 30 seconds
- Display metrics in cards/panels:
  - **System Overview** card: uptime, temperature with color-coded indicator (green < 60°C, yellow < 80°C, red >= 80°C)
  - **CPU / Memory / Disk** cards with progress bars and percentages
  - **Services** card with status indicators (green dot = active, red dot = inactive)
- Fetch data from `GET /health/api` via JavaScript `fetch()`

### Step 4: Ensure Nginx routes `/health` correctly

The nginx config already proxies `/health` to port 5001 (search server). We need to update `deploy/bbc-radio-nginx.conf` to route `/health` to port 5002 (analytics server) where the new dashboard will live. This is a deployment concern — the existing search server also has a `/health` endpoint but the analytics server's health page is more useful.

However, since the user may already have this deployed, we should note this in the plan. The change is a single nginx location block update.

## File Changes

### Modified files

1. **`api/analytics_server.py`**
   - Add `_get_cpu_usage()`, `_get_memory_usage()`, `_get_disk_usage()`, `_get_uptime()`, `_get_cpu_temperature()`, `_get_service_status()` helper functions
   - Add `@app.route('/health/api', methods=['GET'])` JSON endpoint
   - Modify `@app.route('/health', methods=['GET'])` to return an HTML dashboard when browser-accessed (check `Accept` header for `text/html`)

2. **`deploy/bbc-radio-nginx.conf`** (deployment file)
   - Change the `/health` location block to proxy to `http://127.0.0.1:5002` (analytics server) instead of port 5001 (search server)
   - Add `/health/api` location block proxying to port 5002

### New files: none

## Implementation Order

1. Add system metric helper functions to `analytics_server.py`
2. Add `/health/api` JSON endpoint
3. Build the HTML health dashboard at `/health` route
4. Update nginx config to route `/health` to analytics server

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| `/proc/stat` or `/sys/class/thermal` not available on non-Pi systems | All helpers wrapped in try/except, returning `"unavailable"` |
| User doesn't have permission to read systemctl status | Use `systemctl is-active` which works for any user |
| Nginx config change breaks existing search /health | Keep search server /health intact; just re-route to analytics |
| Auto-refresh too frequent on mobile | Use 30s interval, configurable |
